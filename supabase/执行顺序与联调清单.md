# Supabase 执行顺序与联调清单

## 一次性初始化顺序

按下面顺序在 Supabase SQL Editor 中执行：

1. [schema.sql](G:\works\liuli\mommy-kitchen-helper\supabase\schema.sql)
作用：创建 `users`、`babies`、`recipes`、`recipe_collections`、`growth_records` 基础表和索引。

2. [migrations/20260330_add_auth_user_id.sql](G:\works\liuli\mommy-kitchen-helper\supabase\migrations\20260330_add_auth_user_id.sql)
作用：给 `users` 增加 `auth_user_id`、外键、`updated_at` 触发器。
说明：当前版本已调整为可重复执行；如果你之前跑过一部分，再重新执行也不会因为同名约束报错。

3. [migrations/20260421_add_made_recipes.sql](G:\works\liuli\mommy-kitchen-helper\supabase\migrations\20260421_add_made_recipes.sql)
作用：创建 `made_recipes` 表，支持“我做过了”功能。

4. [rls.sql](G:\works\liuli\mommy-kitchen-helper\supabase\rls.sql)
作用：开启 RLS 并为 `users`、`babies`、`recipe_collections`、`growth_records`、`made_recipes`、`recipes` 配置访问策略。

5. [seed.sql](G:\works\liuli\mommy-kitchen-helper\supabase\seed.sql)
作用：导入 8 道核心食谱，支撑食谱列表、首页推荐、周计划与备菜清单。

## 执行后检查

在 Table Editor 中确认存在以下表：

- `users`
- `babies`
- `recipes`
- `recipe_collections`
- `growth_records`
- `made_recipes`

在 `recipes` 表里确认至少有 8 条数据。

在 Authentication 控制台确认：

- 已启用项目 Auth
- 可以正常创建 `auth.users`

## 本地后端启动

在仓库根目录确认 [.env.local](G:\works\liuli\mommy-kitchen-helper\.env.local) 已填写：

- `SUPABASE_URL`
- `SUPABASE_SERVICE_ROLE_KEY`
- `SUPABASE_ANON_KEY`
- `SUPABASE_JWT_SECRET`
- `WECHAT_APP_ID`
- `WECHAT_APP_SECRET`

建议额外确认根目录或后端环境中存在：

- `JWT_SECRET`
- `DEV_LOGIN_ENABLED=true`
- `PORT=3000`
- `CORS_ORIGIN=*` 或你的本地调试域

启动方式：

1. 进入 [backend](G:\works\liuli\mommy-kitchen-helper\backend)
2. 执行 `npm install`
3. 执行 `npm run dev`

正常启动后应看到：

- `Server running on http://localhost:3000`

也可以在 [backend](G:\works\liuli\mommy-kitchen-helper\backend) 目录执行：

`npm run smoke`

它会自动拉起本地服务并验证核心 API。

## 小程序联调

当前小程序请求入口在 [utils/api.js](G:\works\liuli\mommy-kitchen-helper\utils\api.js)：

- 默认请求地址：`http://127.0.0.1:3000/api`
- 登录优先走 `/api/auth/wechat-login`
- 微信登录失败时自动回退 `/api/auth/dev-login`

在微信开发者工具中建议：

1. 打开“本地设置”
2. 勾选“不校验合法域名、web-view（业务域名）、TLS 版本以及 HTTPS 证书”
3. 编译项目
4. 首次进入点击登录
5. 若远端还没有宝宝档案，按引导创建宝宝

## 常见问题

### `made_recipes does not exist`
说明第 3 步迁移没执行，补跑 [20260421_add_made_recipes.sql](G:\works\liuli\mommy-kitchen-helper\supabase\migrations\20260421_add_made_recipes.sql)。

### 食谱列表为空
说明 `seed.sql` 还没执行，或执行失败。先检查 `recipes` 表是否有数据。

### 登录成功但读不到宝宝/生长/收藏
优先检查：

- `users.auth_user_id` 是否已写入
- `rls.sql` 是否已执行
- JWT 是否由后端正确签发

### 执行 `20260330_add_auth_user_id.sql` 报 `users_auth_user_id_unique already exists`
说明这份迁移之前已经部分执行成功。

现在仓库里的新版 [20260330_add_auth_user_id.sql](G:\works\liuli\mommy-kitchen-helper\supabase\migrations\20260330_add_auth_user_id.sql) 已处理成可重复执行。

建议做法：

1. 直接用新版脚本重新执行一次
2. 执行成功后继续下一步 `20260421_add_made_recipes.sql`

### 后端能启动但 Supabase 报认证/权限错误
优先检查：

- `SUPABASE_SERVICE_ROLE_KEY` 是否正确
- 项目 URL 是否对应当前 Supabase 项目
- `.env.local` 是否被后端正确读取

### Supabase 返回 521 / Web server is down
这不是当前仓库代码逻辑错误，而是目标 Supabase 项目当前不可用或临时异常。需要先确认：

- `SUPABASE_URL` 指向的项目仍然存在
- Supabase 项目实例处于可访问状态
- Cloudflare 521 是否已经恢复

在这个状态下，本地 smoke test 会停在 `dev-login` 阶段，因为后端无法访问远端数据库。
